MiniMax-M3 · MXFP4 · TP=1 — gfx1250

Serving-benchmark trace · 1×AMD gfx1250 (warp32, 464 GB) · Triton + Gluon kernel backend, hipBLASLt bf16 fallback · full run (~4552 decode steps). Kernel durations from PyTorch/Kineto MiniMax-M3-MXFP4_ts_20260709_142757_278.pt.trace.gfx1250.json.gz.

This is now a real serving run — not the earlier bring-up. Compared with the previous gfx1250 capture, decode is CUDA-graph captured (CPU-op:GPU-kernel ratio 0.21:1, down from 54:1) and GPU utilization is 76% (373s busy of 488s wall). The sampling / kv-index stall artifacts that dominated the bring-up are gone (now <0.2 s total). Absolute times here are directly meaningful and — after normalizing for the ~10% higher step count — comparable to the gfx950 run.
The story in one line: 71% of GPU time is a single un-tuned hipBLASLt bf16 GEMM (Cijk_…MT32x16x32…ISA1250, 916,971 calls @ 289 µs avg). All bf16 dense projections (aten::mm) fall through to this one poorly-shaped Tensile tile on gfx1250 instead of a tuned Triton GEMM. Fixing that one dispatch is worth more than every other kernel combined.
