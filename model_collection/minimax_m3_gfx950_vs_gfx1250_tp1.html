Same MXFP4 model, single GPU, both full serving runs. gfx950 (warp64), CK ck_tile + hipBLASLt backend. gfx1250 (warp32), Triton + Gluon backend with hipBLASLt bf16 fallback. Both decode-in-CUDA-graphs.

Now an apples-to-apples comparison. Unlike the earlier bring-up capture, the new gfx1250 trace is a real graph-captured serving run at 76% GPU utilization (373s busy / 488s wall, 5.9M kernels). gfx1250 ran ~10% more decode steps than gfx950, so all comparisons below are normalized per decode step (step count derived from the 57-sparse-layer indexer invocation count: gfx950 ≈ 4154 steps, gfx1250 ≈ 4552 steps). Per-step time and per-call latency are both directly meaningful.

Headline: per decode step gfx1250 spends 81.9 ms vs gfx950's 25.7 ms — a 3.19× (219%) gap. But that gap is not broad: it is almost entirely one operation. The bf16 dense-projection GEMM alone is 58 ms/step on gfx1250 vs 5.7 ms on gfx950 (10.2×), and MoE is 1.64×. Every other operation is at parity or faster on gfx1250. Close the dense-GEMM and MoE-GEMM gaps and gfx1250 lands within ~10% of gfx950.

Two targets carry the whole gap:
 ① Dense bf16 GEMM — gfx1250 routes every aten::mm projection to a single un-tuned hipBLASLt tile Cijk_…MT32x16x32…ISA1250 (289 µs avg, one tile for all shapes). gfx950 dispatches the same math to several shape-specialized Tensile tiles (MT32x16x1024, MT256x16x128, MT64x16x512, …) at ~27 µs. This is the 10× gap and 71% of gfx1250 time.
 ② MoE expert GEMM — gfx1250's Triton _moe_gemm_a16w4 (109 µs) + _gemm_a16wfp4 vs gfx950's CK MoeFlatmm — 1.64× and 21% of gfx1250 time — a genuine Triton tuning opportunity.

5 Tuning guidance for gfx1250

Priority 1 — get bf16 dense GEMM off the un-tuned hipBLASLt tile (58 ms/step, 71% of GPU). Every aten::mm projection falls to one Tensile config MT32x16x32 on gfx1250 (all M/N/K share it → 289 µs avg). Two routes, both worth trying:

  • Route a16w16 to a tuned Triton GEMM instead of hipBLASLt — matches the ATOM/AITER gfx1250 GEMM-routing fix (untuned a16w16 → Triton). Even a modestly tuned Triton bf16 tile should beat a single one-size Tensile tile for these skinny decode shapes (M small, N/K = 6144/…).
  • Or re-enable the a8w8 quant path that the earlier bring-up used (_gemm_a8w8_kernel, Triton) — it keeps projections off hipBLASLt entirely. Confirm whether dropping a8w8 here was intentional; running dense proj in bf16 is what created this regression.
  • If staying on hipBLASLt, force per-shape Tensile tuning for gfx1250 (ISA1250) so decode vs prefill shapes get distinct tiles as they do on gfx950.

Priority 2 — tune the Triton MoE expert GEMMs (17 ms/step, 21%). _moe_gemm_a16w4 at 109 µs/call dominates; the decode-shape _gemm_a16wfp4 uses BLOCK_M16×N128×K512, num_warps=8, waves_per_eu=2, matrix_instr_nonkdim=16. On warp32/gfx1250 sweep BLOCK_M (16→32/64 for larger batch), num_warps (4/8), waves_per_eu, and NUM_KSPLIT; gfx950's CK flatmm reaches the same work in 11 ms, so ~1.6× headroom exists.

Priority 3 — attention is near parity (1.14×) but has a long tail. _unified_attention_gluon decode is competitive at ~13 µs; the prefill 2d variant (avg 13 ms) and split-K reduce_segments/reduce_grouped_gluon add up. num_warps=1, waves_per_eu=1 on the gluon kernels looks conservative for gfx1250 — worth a sweep once P1/P2 land.

Do NOT spend time on: norm/RoPE fusion, sparse indexer, activation, embedding, elementwise — §2b shows gfx1250 already runs these 1.4–3.5× faster per call than gfx950. Misc/elementwise is 0.10× (far less on gfx1250 thanks to more fusion). Host/launch overhead is also no longer an issue now that decode is graph-captured.
